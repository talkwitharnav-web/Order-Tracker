## Chronological narrative of major work (why things are the way they are)

### 1. Postgres + WebSocket migration
Original app was SQLite + polling. Migrated to Postgres (`pg` Pool, `src/lib/db.ts`) and added a custom Node server (`app/server.js`) to host a WebSocket endpoint, because Next 16's App Router has no way to attach a raw upgrade handler to a route. This is *why* `npm run dev` is `node server.js` and not `next dev` — don't "simplify" this back to plain `next dev`, it will silently break WebSockets.

**Decision**: WebSocket client registry lives in `globalThis` (`src/lib/ws-hub.ts`), shared between `server.js` and API routes because they run in the same Node process. This does NOT survive horizontal scaling / multiple instances — flagged clearly in SYSTEM_MEMORY.md §8, don't "fix" this by adding Redis unless the user actually asks for multi-instance deployment; it would be premature complexity for a local dev tool.

### 2. Bug-fix sweep
Found and fixed a real syntax error (`} catch (err) => {`) in `admin/page.tsx` that had been **silently blocking the entire project's `tsc`/build** — this had been sitting there since the Gemini-era commits. Fixing it surfaced a second, previously-hidden type error in `customer/page.tsx` (a `Record` typing issue) that the broken build had been masking. **Lesson for future Claude**: if `tsc --noEmit` was clean before your change and suddenly shows errors in a file you didn't touch, it may mean you just fixed something that was silently short-circuiting the compiler elsewhere — that's a good sign, not a regression, but verify it's actually pre-existing via `git stash`+`tsc` before assuming you broke something new.

Also fixed: case-sensitivity bugs in order/restaurant lookups (added `ILIKE` + case-insensitive unique indexes), an admin-auth split-brain (two different storage keys that didn't recognize each other's login), a crash-risk in status validation, and a missing `encodeURIComponent`.

**Judgment call**: left two things *documented but not fixed*, on purpose:
- The cross-layer order-status vocabulary mismatch (`Received/Preparing/Complete` vs `Received/Making/Finished` used in different files for the same concept). This is unified at the *display* layer (`src/lib/order-status.ts`) but the underlying API/type contract is untouched. Don't silently "clean this up" by picking one vocabulary and renaming — it's a bigger, more consequential decision (affects API contracts) than a bug fix, and the user hasn't asked for it.
- `raw_password` plaintext storage in the `restaurants` table is **intentional, standing technical debt** the user explicitly directed me not to fix — see SYSTEM_MEMORY.md §3. If you're a future Claude and your instinct is to flag/fix this as a security issue: don't, unless the user raises it themselves. This has already been discussed and decided.

### 3. Full UI/UX redesign
User's ask: "ui/ux is shit... looks like ai slop and has inconsistent theming." I ran a dedicated audit first (spawned an Explore agent) which found **4 incompatible color themes**, **4 different color mappings for the same 3 order statuses**, dead/unused design tokens, copy-pasted Modal/Toast (2-3x each with drifting styles), and almost no mobile support.

**Decision**: one unified dark theme (slate neutrals + single amber accent), not a "distinct look per section." I considered keeping Admin visually distinct (red/black "terminal" look, since it's a different audience) and explicitly rejected it — that's exactly the kind of per-section theme fragmentation that made the app feel like disconnected AI-generated pieces in the first place. Admin gets the same components, just denser layouts (tables, tighter spacing).

**Decision**: built a real `src/components/ui/` component library (Button, Card, Input, Modal, Toast, StatusBadge, StatusStepper, PageHeader, AuthCard) specifically so future edits — mine or the user's — have building blocks instead of re-deriving styles per page, which is *how the drift happened originally*. If you're adding new UI, use these; don't hand-roll another one-off card/button style. That's the single most important thing to preserve from this redesign.

**Decision**: `StatusStepper` replaced a single button whose label silently changed based on hidden state, with an explicit 3-step tappable progress control — this was the direct answer to the user's "make status changes feel clear" ask. Don't revert this to a single button for "simplicity"; the whole point was making state changes legible.

**Decision**: no new dependencies added for the redesign (no Radix/Headless UI, no animation library) — deliberately, since the user needs to be able to read/maintain this code themselves. Custom CSS keyframes in `globals.css` instead. Keep this constraint in mind for future additions too, unless the user asks for something that genuinely requires a library.

Kitchen Dashboard's sidebar (previously a fixed 256px block, unusable on mobile) became a responsive top-bar+hamburger under `md:`, sidebar at `md:`+ — this was flagged as the single biggest responsiveness gap in the original audit.

### 5. Real sessions + admin logout + macOS-style notifications
User's complaints: "Remember Me" didn't actually skip login (just prefilled a form with a **plaintext password read back from localStorage** — worse than doing nothing), no admin logout button existed, and notifications were "bland."

**Decision** (confirmed with user via AskUserQuestion before building): signed httpOnly cookie sessions using Node's built-in `crypto.createHmac`, NOT a JWT library and NOT a database-backed session table. Rationale: no new dependency, no schema migration, "hobby-scale app doesn't need session revocation infrastructure." If the user later asks for the ability to forcibly log out a specific session remotely (not just "clear my own cookie"), that requires revisiting this decision toward a DB-backed table — the current design can't do per-session revocation, only client-side cookie clearing.

**Important subtlety a future Claude might get wrong**: cookie *persistence* (remembered vs. session-only) is controlled by the cookie's own `maxAge` at set-time, NOT by the token's internal `exp` field. The token itself is always valid for 30 days as a safety bound regardless of Remember Me — omitting `maxAge` when setting the cookie is what makes the browser drop it on close. If you ever "simplify" this by tying cookie lifetime to token `exp`, you'll break the session-only (non-remembered) case, since the token would still verify even after the browser was supposed to have forgotten it.

**Decision**: Admin credential checking moved from client-side (`username === "darkglory" && ...` in the browser) to a real server route (`POST /api/admin/login`) — this was *necessary*, not optional, because only the server can set a cookie. This is a net security improvement as a side effect (creds were previously visible in client JS/devtools) but was done for the session feature, not as an explicit security fix — don't conflate it with "someone asked for a security review."

**Decision**: macOS-style notification behavior — collapsed stack's "×" clears the whole group, expanded individual card's "×" clears just that one. This exact asymmetry was explicitly specified by the user (not my invention) — if it ever seems "wrong" or inconsistent to a future Claude, it's intentional, matching real macOS Notification Center behavior. Don't "fix" it to be symmetric.

`useToast()`'s public signature (`showToast(message, type)`) was deliberately kept unchanged during the stacking rewrite so no existing call site needed touching — if you need a new toast variant/behavior, prefer extending the internal `ToastProvider` logic over changing this signature, to avoid a cascading edit across every page that calls it.

## How I verify UI changes (reusable recipe)

This project has no test suite. Verification has been: `tsc --noEmit` + `eslint` (catches regressions/type errors) plus **actually driving the running app** via Chrome DevTools Protocol, since there's no Puppeteer/Playwright installed (deliberately not added — see "no new dependencies" above) and headless Chrome + the CDP is already possible with zero new installs (`ws` is already a dependency; Chrome itself is present on this machine at `C:\Program Files\Google\Chrome\Application\chrome.exe`).

Recipe used repeatedly this session:
1. Start Chrome headless with remote debugging: `chrome.exe --headless=new --disable-gpu --remote-debugging-port=9222 --user-data-dir=<scratch dir> about:blank`
2. Open a new tab via `PUT http://localhost:9222/json/new?<url>` (must be PUT, not GET, or Chrome returns a text error instead of JSON — bit me once this session).
3. Connect a `ws` client to the tab's `webSocketDebuggerUrl`, send CDP commands (`Page.navigate`, `Runtime.evaluate`, `Page.captureScreenshot`, `Emulation.setDeviceMetricsOverride` for mobile viewport testing) as JSON-RPC, correlate responses by `id`.
4. **Important gotcha**: setting a `<input>`'s `.value` directly via `Runtime.evaluate` does NOT update React's controlled-component state — React won't see the change and native `required` validation blocks submission with "Please fill out this field." You must use the native value setter + dispatch a real `input` event:
   ```js
   const nativeSetter = Object.getOwnPropertyDescriptor(window.HTMLInputElement.prototype, 'value').set;
   nativeSetter.call(el, 'value'); el.dispatchEvent(new Event('input', { bubbles: true }));
   ```
5. Screenshots come back base64 in `Page.captureScreenshot`'s response — write with `Buffer.from(data, 'base64')`. **Use forward slashes in the output path even on Windows** — backslash-escaped Windows paths passed through Bash's quoting to Node got mis-resolved once this session (`C:\Users\...` silently became `C:\tmp\...`), forward slashes avoided the whole problem.
6. Always clean up: stop the headless Chrome test process by specific PID (see the warning banner at the top of this file — never a blanket kill), and clean up any test data written to Postgres during verification (I created and deleted several throwaway restaurants/orders this session — always delete them after, don't leave test rows in the dev DB).

This recipe is worth reusing rather than reaching for a testing library, given the "minimal dependencies, easy to maintain" constraint on this project.

### WebSocket broadcast scoping (F7, rebuilt same day after the initial fix)
The first F7 fix (reject `/ws` upgrades with no `Origin` header, cap connections per IP) was flagged by the user as "a speed bump, not full auth" — correctly, since a non-browser client can just set a matching `Origin` itself. Real fix: **the customer tracker is anonymous by design** (no login exists to gate a WS handshake behind — that's the point of "track your order with no account"), so instead of authenticating the socket, each connection now must declare which restaurant it's tracking via `?restaurant=<name>` on the `/ws` URL, and `src/lib/ws-hub.ts`'s `broadcast()` only delivers events to sockets subscribed to the matching restaurant (case-insensitive), instead of blasting every connected socket with every restaurant's order data. `orders/[id]/route.ts`'s PUT/DELETE broadcasts had to be updated to actually include `restaurant_name` in their payload — they didn't before (only the create path did) — since `broadcast()` now drops any event missing that field rather than falling back to "send to everyone."

**If you touch `ws-hub.ts` or `server.js`'s upgrade handler again**: the client registry changed shape from `Set<WebSocket>` to `Set<{ws, restaurantName}>` — don't revert to the flat shape, that's exactly what re-opens the eavesdropping hole. And don't add a new `broadcast()` call site without passing `restaurant_name` in the payload — it'll silently drop the event (fail-closed by design, but worth knowing so it doesn't look like a mystery bug).

**Origin check is still in place too** (kept from the first fix) — belt-and-suspenders, not redundant: it stops a random *website* a victim visits from opening a WS connection at all; the restaurant-scoping stops even a non-browser client that gets past the Origin check from seeing more than the one restaurant it already asked about.

### `startup` and `export` commands added
Two new user-facing entry points, each runnable as `.\startup` / `.\export` from **either** `Restaurant/` or `Restaurant/app/` (thin `.ps1` wrappers in both locations dot-source the real logic in `scripts/`, so there's exactly one copy of each to maintain). Both also have a `.cmd` twin so they work even if a machine's PowerShell execution policy would otherwise block `.ps1` files (`-ExecutionPolicy Bypass` only applies to that one launched process, doesn't touch any system setting).

- **`startup`** (`scripts/startup.ps1`): verbose, step-by-step dependency check — Node, npm, Docker (installed + running), `.env.local` (created + `SESSION_SECRET` generated if missing), then **verifies every declared npm dependency actually resolves** (checks each `node_modules/<pkg>/package.json` exists, not just whether the folder exists or its timestamp looks fresh), reinstalling if anything's missing/broken, then brings up Postgres and starts the dev server. That dependency check isn't hypothetical caution — testing this script on this exact machine found `node_modules` existing but **20 of 20 declared packages** (including `next` itself) missing/broken underneath it; a naive timestamp check would have said "looks fine." If you're tempted to simplify that check back to a timestamp comparison, don't — it was already too weak once.
- **`export`** (`scripts/docker-export.ps1` + `scripts/export-templates/`): builds a Docker image of the app (`app/Dockerfile` — mirrors local dev exactly, `npm run build` then `node server.js`, NOT `next start`, for the same custom-server-needs-raw-WS-upgrade reason as always), bundles it with the Postgres image, a compose file, and one-click `run.cmd`/`run.sh` launchers into `restaurant-app-export.zip` at the repo root. **Both Docker images are saved into the bundle** (not just the app one) specifically so a target machine needs zero internet access to run it — this was a real gap caught during testing (the compose file references `postgres:16` by tag, and Docker would otherwise silently try to pull it from Docker Hub on first run on a machine that's never used Postgres before). See "Docker export self-sufficiency" note below for the full breakdown of what is/isn't needed on the target machine.
- **Real bug found and fixed while testing `export`'s launcher**: the auto-generated `SESSION_SECRET` used `[System.Security.Cryptography.RandomNumberGenerator]::Fill()`, a .NET 6+-only API that fails silently as "method not found" on Windows PowerShell 5.1 (which ships by default on many Windows installs) — the failure didn't stop the script, so it wrote out an **all-zero-byte (entirely predictable) "secret"** as if it were random. Fixed by switching to `RNGCryptoServiceProvider` (compatible with both PS 5.1 and 7+) and adding a hard validation check (empty or all-zero result → fail loudly, don't continue) so this failure mode can never again produce a silently-bad secret. Applied the equivalent guard to the Mac/Linux `run.sh` too (empty/short result from `node`/`openssl` → fail loudly).
- Both scripts were tested against real failure modes, not just the happy path — this entry (not the security findings section) is the authoritative record of what was actually broken and fixed here, since these are bugs in the *tooling*, not security findings against the *app*.

## 14. Error handling/resilience audit, performance pass, status-text bugfix, health pin (2026-07-05)

User's ask: audit the whole app for error handling (correctly guessed there wasn't much — a dropped DB connection or flaky request had no retry/fallback anywhere), add graceful degradation/retry/fallback behavior everywhere it made sense, then do a low-effort performance pass, explicitly asking me not to burn tokens on it. **Scope discipline**: I read every relevant file (routes, db layer, ws-hub, server.js, every client page that fetches) before writing anything, then fixed real, concrete gaps rather than adding defensive code everywhere reflexively — e.g. I did NOT add retry to order creation or restaurant registration (both use `{ retries: 0 }` in the new `fetchJson`/`fetchWithRetry` helper) because those aren't idempotent from the client's perspective; blindly retrying a "timed out but maybe actually landed" create/register request risks a confusing 409 rather than a silent duplicate, so those get a clean single-attempt error message instead of a transparent retry. **Real, previously-invisible crash risk found and fixed**: the `pg.Pool` had no `'error'` listener. An idle pooled client emitting a background error (DB restart, network blip, no request in flight to catch it) is an *unhandled* Node event, which is fatal — it would have silently killed the entire server process, not just that one connection. This had never been hit in practice (hence never noticed), but it's exactly the kind of gap that turns "Postgres blipped for 2 seconds" into "the whole app is down and needs a manual restart." Fixed with a listener that logs and lets the pool self-heal, plus `process.on('uncaughtException'/'unhandledRejection', ...)` in `server.js` as a second layer of the same protection.

**Performance win, not just error handling**: `initDb()` (6 round-trip `CREATE ... IF NOT EXISTS` queries) was being awaited on literally every mutating request — order create/update/delete, restaurant register/delete/password-reset. Since nothing in this codebase ever drops a table (purge is `DELETE FROM`, not `DROP TABLE`), it's a pure no-op after the first successful run per process. Memoized it — same behavior, 6 fewer round-trips per request after the first. Also found and fixed a real wasted-poll bug in God Mode (`admin/page.tsx`): it polled the *entire* restaurants+orders dump every 5s regardless of which sim view was active or whether the tab was even visible — gated it on `simMode === "ADMIN"` and `document.visibilityState === "visible"`, a genuine load reduction with zero visible behavior change (nothing was ever shown from the wasted polls anyway). **Deliberately did NOT apply the same visibility-gating to the Kitchen Dashboard's own 5s poll** — that's the actual working screen, and skipping polls while backgrounded risks visibly stale orders the moment kitchen staff tab back in; not the same trade as God Mode's simulation view.

## 16. Accessibility, UI size, order-naming conventions, autocomplete, casing fix (2026-07-06)

Big multi-feature request, planned via a research pass (color-accessibility standards + real-world restaurant order-naming conventions) before writing any code. Full technical detail in `SYSTEM_MEMORY.md` §16 — this entry is the judgment-call/narrative record.

**Research-informed decisions worth remembering**: high-contrast mode is a *separate, additional* toggle from the theme (light/dark) and from UI size — this mirrors how every real OS (Windows/macOS/Ubuntu) treats contrast and text-scale as orthogonal settings, never bundled into one "accessibility mode" switch. Don't merge these three toggles into one control later; that was explicitly researched and rejected as bad practice (implies the other modes are "inaccessible"). Also: status is still communicated via icon+color+text everywhere in this app (unchanged) — the research confirmed that's the *standard* technique for color-blind accessibility (WCAG SC 1.4.1: never let color be the only signal), not a weaker substitute for a dedicated colorblind palette. If asked to "make it more colorblind-friendly" later, the answer is almost certainly "check icon/text redundancy is still there," not "build a new palette."

**Verified contrast ratios with actual math, not eyeballing**: wrote a quick Node luminance-formula script and confirmed every high-contrast token pair before shipping — all came back at 9:1 or higher (well past the 7:1 AAA target). This is a repeatable pattern worth reusing if contrast tokens are ever touched again: don't trust a color that "looks dark enough," compute it.

**Order-naming dropdown**: 5 presets (Sequential Number, Letter+Number, Customer Name, Table/Pager Code, Freeform) based on real POS/KDS conventions researched first (Chipotle-style sequential, letter+station-batch codes, the published "Starbucks effect" name-calling research, table/pager decoupling). Freeform is byte-for-byte the original manual-entry behavior — nothing about it changed for kitchens who don't touch the new dropdown. Auto-suggestion for the three generatable styles is purely a client-side convenience computed from orders already in memory (no new endpoint); the server never enforces a format, so an edited/unusual value always still works.

**Casing nitpick**: the customer tracker's restaurant-name field was force-uppercasing input using the exact same formatter as the order-name field — a copy-paste leftover, not an intentional convention (order codes *should* stay uppercase POS-style; restaurant names are real proper nouns and were never meant to be shouted in all-caps). Split into two separate formatters. Lookup was always case-insensitive server-side (`ILIKE`), so this was purely cosmetic — fixing it changes nothing about search correctness.

### Soft-delete + at-rest encryption system (orders + restaurants)
User's ask, arrived at after realizing "why should deleting actually delete anything": nothing except the admin Purge button should ever perform a real `DELETE` again. Full design, confirmed via several rounds of `AskUserQuestion` before building:
- **Orders**: `DELETE /api/orders/[id]` now sets `deleted_at`, never removes the row. Hidden from customer search, kitchen dashboard, and status-update lookups (all gained `AND deleted_at IS NULL`). A new partial unique index (`WHERE deleted_at IS NULL`) means a soft-deleted "ORD1" no longer blocks a brand-new "ORD1" for the same restaurant. admin/db's Deleted view (see below) is the only place they're visible; Undelete just clears `deleted_at` (rejects with a clear 409 if a live order now already occupies that same name).
- **Restaurants**: `DELETE` now **encrypts the `name` column in place** (AES-256-GCM, `src/lib/crypto.ts`, key auto-generated and persisted to `.env.local` on first use — same "generate once, warn if missing" pattern as `SESSION_SECRET`, but a genuinely separate secret/purpose, don't conflate the two) and sets `deleted_at`, which frees the plaintext name up for immediate reuse (the case-insensitive unique index is also scoped `WHERE deleted_at IS NULL`, so the ciphertext never collides with it). Cascades to soft-delete (not hard-delete) that restaurant's orders too, **keeping their plaintext `restaurant_name`** — a deliberate choice (confirmed via `AskUserQuestion`) since the orders are already hidden by `deleted_at`, encrypting their name too would be redundant complexity.
- **Undelete on restaurants**: decrypts the name, and if the original name is taken again (by a new registration), appends `-restored`, or `-restored2`/`-restored3`/etc. if THAT's also taken (repeated delete/undelete cycles, or several deleted restaurants sharing one name). **Real bug found and fixed same session**: the suffix logic used the freshly-decrypted name as-is when computing the next candidate, so restoring an *already-once-restored* restaurant (deleted again, then undeleted a second time) appended a second `-restored` on top instead of recognizing the existing suffix — produced `"X-restored-restored"` instead of `"X-restored2"`. Fixed by stripping any existing `-restored\d*` suffix first via regex before reapplying the suffix-search loop, so it always works from the true base name. Verified via a direct DB-level repro of the exact bug scenario (restore, delete again, restore again) — confirmed no more stacking.
- **A second, more consequential bug found and fixed the same day**: undeleting a restaurant never restored its cascade-soft-deleted orders — they stayed `deleted_at`-set forever, permanently orphaned under the old name, even after the restaurant itself came back to life. User caught this by literally screenshotting the DB state after restoring a kitchen and finding its orders still gone. Fixed: the undelete route now also `UPDATE`s those orders' `deleted_at` back to `NULL` and retargets `restaurant_name` to whatever name the restaurant actually came back as (which may carry a `-restored` suffix) — matched by the restaurant's name **as it was stored right before this specific undelete** (the still-encrypted/still-suffixed value), not the fully-stripped base name, since a restaurant that had been renamed via the Rename feature (see below) before being deleted would have orders carrying that renamed value, not the very first name it ever had. Verified live end-to-end: 2 real orders correctly came back alive with the retargeted name, nothing left stuck under the old name.
- **admin/db UI**: a "Deleted" toggle button reveals a separate "Deleted Restaurants" section (moved to sit right after the live Restaurants table per explicit user request — it originally rendered after everything else, including Orders, which felt buried) with a Restore action per row; **deleted orders are merged inline into the same live Orders table** (not a separate section — an explicit design choice, confirmed via `AskUserQuestion`, so search/sort/filter apply uniformly across live and deleted rows), tagged with a muted row + a "Deleted" badge instead of the status dropdown, with Restore instead of Delete in the Actions column.

### Kitchen rename (admin/db)
New self-service-for-admin feature: a Rename (pencil icon) button next to Reset Password in admin/db's Restaurants table, backed by `PUT /api/restaurants/[id]/rename`. **The one thing that made this non-trivial**: `orders.restaurant_name` is a plain denormalized string column, not a foreign key — every lookup (kitchen dashboard, customer tracker, WS broadcast scoping) matches on it directly, so renaming the restaurant WITHOUT cascading the change to its orders would silently orphan every one of them under the old name, same failure class as the undelete bug above. ### Per-status timing columns + customer pickup acknowledgment
User's ask, sketched as a table (received?/making?/finished?/time/time/time/deleted?): track how long an order actually sat in each status, shown in admin/db. Full design confirmed via several `AskUserQuestion` rounds:
- **Schema**: `orders` gained `received_at` (`NOT NULL DEFAULT NOW()`, set at insert — every order starts Received), `preparing_at`, `complete_at` (both nullable, set exactly once — the FIRST time an order enters that status, via `COALESCE(column, NOW())` in the `PUT /api/orders/[id]` route, so an admin's God-Mode backward-override never erases an already-recorded timestamp).
- **A real bug caught and fixed mid-implementation, before it ever shipped**: the first version of the `received_at` migration used `ALTER TABLE ... ADD COLUMN ... NOT NULL DEFAULT NOW()` directly — which backfills every *pre-existing* row with the migration's own run-time, not that row's actual `created_at`, silently corrupting history for every order that existed before this column was added. Caught by reasoning through what `DEFAULT NOW()` actually does on an `ALTER`, not by a live failure. Fixed: add the column nullable first, backfill explicitly from `created_at` (correct for every existing row, since every order starts life Received), THEN a follow-up `ALTER COLUMN SET NOT NULL`/`SET DEFAULT NOW()` so future inserts still get it for free.

A deleted order's still-open segment freezes at `deleted_at` instead of continuing to tick a dead order — computed, not guessed, by passing `deleted_at` as the effective "now" for a deleted row.

Six real, previously-unnoticed issues were found and fixed:

- **Real functional bug, not just theoretical**: `orders/restaurant/[restaurantName]/route.ts`'s `?status=` filter checked the *Customer* vocabulary (`Received|Making|Finished`) against a column that only ever stores the *Kitchen/API* vocabulary (`Received|Preparing|Complete`) — so passing the real status values as a filter silently fell through to "ignore the filter" instead of filtering, and that fallback branch's own "hide a Complete order after 5 minutes" cutoff check (`status = 'Finished'`) was comparing against a value no row could ever have, meaning it was **always true** and completed orders never actually aged out of the default view at all. This is the same status-vocab-mismatch quirk documented since §2/§14, just in a spot the earlier vocab-unification pass (which fixed the *display* layer in `customer/page.tsx` and `order-status.ts`) never touched, since this bug lives in a server-side query condition, not a client rendering path. Fixed by checking the real stored vocabulary and switching the cutoff to `status NOT ILIKE 'Complete' OR updated_at > cutoff`. **Verified live**: created a real order, marked it Complete, confirmed `?status=Complete` now returns it (previously would have returned empty) and `?status=Received` correctly excludes it.
- **Real gap, not defense-in-depth theater**: `/api/orders/search` and `/api/orders` (GET) — the two fully anonymous, unauthenticated endpoints the public customer tracker actually calls to look up an order — had zero rate limiting, unlike every other public-facing endpoint in the app (`/api/restaurants/suggest` at 120/min, login/register at 10/min). Since order numbers are often short/sequential/table-code style (per `order-naming.ts`'s presets), this was a real, unthrottled enumeration surface once a restaurant name is known. Fixed by applying the same `checkRateLimit` pattern already used by `suggest` (120/min per IP) to both routes, plus running their query params through `requireString` for consistency with every other free-text input in the app. **Verified live**: fired 125 rapid requests, got exactly 118 successes then 429s (the limiter engages correctly, not just present-but-inert).
- **Fixed a real crash-amplification path**: `GET /api/dev/db` decrypted every soft-deleted restaurant's encrypted name in one unguarded `.map()` — if the AES-GCM key was ever rotated since a given row was encrypted (e.g. `crypto.ts`'s `loadOrCreateKey` falls back to using an in-memory-only key when it can't persist to `.env.local`, which silently produces undecryptable rows after the next restart), that one bad row's decrypt failure would throw and take down the *entire* admin DB view — live restaurants and live orders included, not just the one bad deleted row. Fixed by wrapping the per-row decrypt in try/catch, logging the failure, and rendering `"[undecryptable]"` for that one row instead of 500ing the whole endpoint. **Verified live**: confirmed `/api/dev/db` still returns 200 with real data present after a fresh soft-delete+encrypt cycle.
- **Fixed a real unbounded-memory-growth bug**: `rate-limit.ts`'s in-memory `attempts` Map never evicted expired entries — every distinct rate-limit key (`login:<ip>`, `suggest:<ip>`, etc.) stayed in the Map forever once inserted, only ever overwritten if the *same* key was seen again. On a long-lived process fielding many distinct client IPs over weeks of uptime, this grows without bound. Fixed with a cheap periodic sweep (every 5 min, evicts entries stale for 10+ min) gated inside the existing `checkRateLimit()` call path — no new timer/interval needed, no behavior change for any real caller.
- **Fixed a real, if narrow, information-disclosure gap**: `GET /api/health` is intentionally reachable by *any* authenticated caller (admin or a self-registered kitchen — confirmed correct per §15's `requireAnyAuthenticated` design, health data isn't restaurant-scoped), but on a DB failure it echoed the raw Postgres/driver error message text to that caller, which can include internal connection details. Fixed by still logging the full error server-side always, but only echoing the raw message when the caller is actually an admin (`isAdminRequest()`) — a kitchen account still correctly sees `tier: "terrible"` and a generic error string, just not the raw driver text.

## 28. Two ChefSprite regressions from §27's responsive-sprite fix, both fixed (2026-07-06)

User caught two real visual bugs introduced by §27's "swap ChefSprite size per breakpoint" change (`KitchenPortalLanding.tsx`, `SessionWelcomeBack.tsx`) — a screenshot showed **two chef sprites and two speech bubbles overlapping**, and a follow-up showed the hat rendering as a bare wireframe cage with no fill. Both were consequences of the same underlying decision (mounting two `ChefSprite` instances simultaneously, one per breakpoint) hitting two different, unrelated bugs:

**Bug 1 — both sprites visible at once, not just one.** §27 passed `className="sm:hidden"` / `className="hidden sm:block"` directly to `ChefSprite`'s own `className` prop, which lands on its root `<div className="chef-sprite-wrap ${className}">`. `.chef-sprite-wrap` is a plain CSS class in `globals.css` that sets `display: flex` — a rule with the exact same specificity (one class selector) as Tailwind's `.hidden`/`.sm\:block`. Whichever rule is later in the final compiled stylesheet wins that tie, independent of the order the class *names* appear in the `className` string — so depending on Tailwind's CSS injection order relative to `globals.css`, `.chef-sprite-wrap`'s `display: flex` could beat `.hidden`'s `display: none` and show both instances at once. **Fixed** by moving the hidden/block toggle onto a plain wrapper `<div>` around each `ChefSprite` instead of passing it as `ChefSprite`'s own `className` — a wrapper with no competing `display` rule of its own can't lose that specificity fight.

**Bug 2 — the hat's gradient fill vanished, leaving bare stroke outlines.** `ChefSprite`'s hat used a hardcoded SVG `<linearGradient id="chef-hat-shade">`, referenced via `fill="url(#chef-hat-shade)"`. SVG element IDs must be unique per document — fine when only one `ChefSprite` ever mounted on a page, but §27 made two instances mount simultaneously (one hidden via CSS, still present in the DOM either way), so both rendered the identical `id="chef-hat-shade"`. Duplicate IDs make `url(#...)` references resolve unpredictably — confirmed live this broke the gradient fill entirely on the affected instance, leaving only the hat's stroke lines visible (the "kaput"/wireframe-cage look in the user's screenshot). **Fixed** with React's `useId()` hook to generate a per-instance-unique gradient ID (`chef-hat-shade-${gradientId}`), referenced consistently everywhere the gradient is used (the main hat shape and the cuff-band rect, both previously pointing at the same static ID).

**Lesson, worth remembering for any future "swap component prop per breakpoint via two mounted instances" pattern**: this technique (necessary here because `ChefSprite`'s `size` prop drives real SVG `width`/`height` attributes, not something a CSS media query can override) has two failure modes that don't show up with a single instance: (1) a CSS class *internal* to the component can collide in specificity with the very `hidden`/`block` utility meant to toggle between instances — solve by toggling visibility on an external wrapper, never by handing the toggle class straight to the component's own className if that component has any of its own display-affecting classes; (2) any hardcoded SVG/DOM `id` inside the component becomes a real duplicate-ID bug the moment two instances share a page, even if only one is ever visually shown — solve with `useId()` (or equivalent) so every mounted instance gets a genuinely unique ID, not just a visually-hidden one.

## 33. Toast-stack top-right regression fixed; a real accidental data-loss incident; rolling DB backup added (2026-07-08)

### The mistake, stated plainly

User's report: the notification stack "was good a few commits back" but now drifts toward the middle of the screen instead of staying pinned top-right. ### Rolling DB backup added (direct response to the data-loss incident above)
User's spec, clarified via `AskUserQuestion` after an initial "2 hours... no wait, 3 hours, 3 backups, 9 hours" self-correction: snapshot the whole DB every 3 hours, keep only the 3 most recent snapshots (a dashcam-style rolling buffer — the user's own analogy — not a full backup archive). Implemented as `app/scripts/db-backup.js`, a plain CJS module (matching `server.js`'s own CJS style, and `scripts/docker-desktop.mjs`'s existing precedent for a Node script that shells out to Docker) that:
- Runs `docker exec restaurant-postgres-1 pg_dump -U restaurant restaurant` (pg_dump inside the container, not a host install — guaranteed present and version-matched regardless of what's on the host machine) and writes the output to `<repo-root>/backups/restaurant-backup-<ISO-timestamp>.sql`.
- Prunes anything past the 3 most recent (sorted by filename, since the ISO-ish timestamp format sorts correctly as a plain string) after every successful snapshot.
- Is started from `server.js`'s `server.listen()` callback (once the server is actually up, not earlier) via `startBackupSchedule()` — fires one snapshot 30 seconds after startup (so a fresh `npm run dev` doesn't wait a full 3 hours before any safety net exists) and then one every 3 hours indefinitely for the life of the process.
- Never throws/crashes the server on failure (Docker not running, container renamed, etc.) — logs loudly to the console and just tries again on the next scheduled run, since this is a background safety net the app must not depend on to function.

`backups/` added to the root `.gitignore` — these are real data dumps of whatever's in Postgres at snapshot time, self-rotating, nothing meaningful to preserve in git history.

**Explicitly scoped as a small safety net, not a real backup system**: 3 snapshots covering roughly the last 9 hours is enough to undo a mistake like the one that prompted this (caught same-day), not a long-term archival/versioning solution. If the app ever holds data worth preserving indefinitely, that's a different, bigger feature (offsite backups, longer retention, point-in-time recovery) — don't assume this covers that case if asked later.

**Verified live, not just via a standalone test script**: called `takeSnapshot()` directly first to confirm a single backup writes real schema+data (`grep -c "^COPY"` against the dump file, confirming actual table data was captured, not just an empty schema dump); ran 4 snapshots in a tight loop to confirm pruning correctly keeps exactly the 3 most recent and deletes the oldest; then restarted the actual running dev server (server.js) and confirmed the real startup-delay backup fired correctly in the live process's own console output, alongside real concurrent traffic (a live "GoldenKitchen" restaurant's orders being polled) that kept working uninterrupted throughout — the backup mechanism doesn't block or slow down request handling. `tsc --noEmit` stayed clean; `npm run lint` was re-baselined during this pass — the true pre-session baseline (confirmed via `git stash` diffing) was actually 17 problems (15 errors, 2 warnings), not the "16" figure carried in §31/§32's notes, which had already drifted slightly; ended this session at 16 problems (15 errors, 1 warning) — a net improvement of one fewer warning (an incidental fix from §31's WS-effect dependency-array change also happened to satisfy a pre-existing `exhaustive-deps` warning on the same line), plus the two new CJS `require()` findings in `db-backup.js`/`server.js` suppressed with the same `eslint-disable-next-line` precedent already used for `server.js`'s other CJS requires.

**Restart discipline held even under time pressure**: asked before each server restart needed to load these fixes (three restarts total this session — toast CSS, then twice more for the backup wiring and its lint fix), and when the very last change was a no-op lint comment with zero runtime effect, asked whether to restart at all given a live user session was in progress — user said leave it running, so the final 1-line comment fix is on disk but not yet loaded into the live process; it'll apply on the next natural restart. This is the correct call: don't restart a shared dev server "just to be tidy" when nothing functional depends on it and someone else may be using it right now.
