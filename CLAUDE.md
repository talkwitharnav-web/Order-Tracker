## Chronological narrative of major work (why things are the way they are)

### 1. Postgres + WebSocket migration
Original app was SQLite + polling. Migrated to Postgres (`pg` Pool, `src/lib/db.ts`) and added a custom Node server (`app/server.js`) to host a WebSocket endpoint, because Next 16's App Router has no way to attach a raw upgrade handler to a route. This is *why* `npm run dev` is `node server.js` and not `next dev` — don't "simplify" this back to plain `next dev`, it will silently break WebSockets.

**Decision**: WebSocket client registry lives in `globalThis` (`src/lib/ws-hub.ts`), shared between `server.js` and API routes because they run in the same Node process. This does NOT survive horizontal scaling / multiple instances — flagged clearly in SYSTEM_MEMORY.md §8, don't "fix" this by adding Redis unless the user actually asks for multi-instance deployment; it would be premature complexity for a local dev tool.

### 2. Bug-fix sweep
Found and fixed a real syntax error (`} catch (err) => {`) in `admin/page.tsx` that had been **silently blocking the entire project's `tsc`/build** — this had been sitting there since the Gemini-era commits. Fixing it surfaced a second, previously-hidden type error in `customer/page.tsx` (a `Record` typing issue) that the broken build had been masking. **Lesson for future Claude**: if `tsc --noEmit` was clean before your change and suddenly shows errors in a file you didn't touch, it may mean you just fixed something that was silently short-circuiting the compiler elsewhere — that's a good sign, not a regression, but verify it's actually pre-existing via `git stash`+`tsc` before assuming you broke something new.

Also fixed: case-sensitivity bugs in order/restaurant lookups (added `ILIKE` + case-insensitive unique indexes), an admin-auth split-brain (two different storage keys that didn't recognize each other's login), a crash-risk in status validation, and a missing `encodeURIComponent`.

**Judgment call**: left two things *documented but not fixed*, on purpose:
- The cross-layer order-status vocabulary mismatch (`Received/Preparing/Complete` vs `Received/Making/Finished` used in different files for the same concept). This is unified at the *display* layer (`src/lib/order-status.ts`) but the underlying API/type contract is untouched. Don't silently "clean this up" by picking one vocabulary and renaming — it's a bigger, more consequential decision (affects API contracts) than a bug fix, and the user hasn't asked for it.
- `raw_password` plaintext storage in the `restaurants` table is **intentional, standing technical debt** the user explicitly directed me not to fix — see SYSTEM_MEMORY.md §3. If you're a future Claude and your instinct is to flag/fix this as a security issue: don't, unless the user raises it themselves. This has already been discussed and decided.

### 3. Full UI/UX redesign
User's ask: "ui/ux is shit... looks like ai slop and has inconsistent theming." I ran a dedicated audit first (spawned an Explore agent) which found **4 incompatible color themes**, **4 different color mappings for the same 3 order statuses**, dead/unused design tokens, copy-pasted Modal/Toast (2-3x each with drifting styles), and almost no mobile support.

**Decision**: one unified dark theme (slate neutrals + single amber accent), not a "distinct look per section." I considered keeping Admin visually distinct (red/black "terminal" look, since it's a different audience) and explicitly rejected it — that's exactly the kind of per-section theme fragmentation that made the app feel like disconnected AI-generated pieces in the first place. Admin gets the same components, just denser layouts (tables, tighter spacing).

**Decision**: built a real `src/components/ui/` component library (Button, Card, Input, Modal, Toast, StatusBadge, StatusStepper, PageHeader, AuthCard) specifically so future edits — mine or the user's — have building blocks instead of re-deriving styles per page, which is *how the drift happened originally*. If you're adding new UI, use these; don't hand-roll another one-off card/button style. That's the single most important thing to preserve from this redesign.

**Decision**: `StatusStepper` replaced a single button whose label silently changed based on hidden state, with an explicit 3-step tappable progress control — this was the direct answer to the user's "make status changes feel clear" ask. Don't revert this to a single button for "simplicity"; the whole point was making state changes legible.

**Decision**: no new dependencies added for the redesign (no Radix/Headless UI, no animation library) — deliberately, since the user needs to be able to read/maintain this code themselves. Custom CSS keyframes in `globals.css` instead. Keep this constraint in mind for future additions too, unless the user asks for something that genuinely requires a library.

Kitchen Dashboard's sidebar (previously a fixed 256px block, unusable on mobile) became a responsive top-bar+hamburger under `md:`, sidebar at `md:`+ — this was flagged as the single biggest responsiveness gap in the original audit.

### 4. Docker Compose for local Postgres
Added `docker-compose.yml` at repo root + `npm run db:up/down/stop/start:all` scripts, runnable from either repo root or `app/` (root `package.json` forwards via `npm --prefix app`). **Note**: mid-session, Docker Desktop was found "manually paused" — I do not ever auto-unpause or restart Docker Desktop myself; I ask the user to do it. This happened once already in this session; if it happens again, ask again, don't assume it's fine to just wait it out or try to control Docker Desktop's process directly.

### 5. Real sessions + admin logout + macOS-style notifications
User's complaints: "Remember Me" didn't actually skip login (just prefilled a form with a **plaintext password read back from localStorage** — worse than doing nothing), no admin logout button existed, and notifications were "bland."

**Decision** (confirmed with user via AskUserQuestion before building): signed httpOnly cookie sessions using Node's built-in `crypto.createHmac`, NOT a JWT library and NOT a database-backed session table. Rationale: no new dependency, no schema migration, "hobby-scale app doesn't need session revocation infrastructure." If the user later asks for the ability to forcibly log out a specific session remotely (not just "clear my own cookie"), that requires revisiting this decision toward a DB-backed table — the current design can't do per-session revocation, only client-side cookie clearing.

**Important subtlety a future Claude might get wrong**: cookie *persistence* (remembered vs. session-only) is controlled by the cookie's own `maxAge` at set-time, NOT by the token's internal `exp` field. The token itself is always valid for 30 days as a safety bound regardless of Remember Me — omitting `maxAge` when setting the cookie is what makes the browser drop it on close. If you ever "simplify" this by tying cookie lifetime to token `exp`, you'll break the session-only (non-remembered) case, since the token would still verify even after the browser was supposed to have forgotten it.

**Decision**: Admin credential checking moved from client-side (`username === "darkglory" && ...` in the browser) to a real server route (`POST /api/admin/login`) — this was *necessary*, not optional, because only the server can set a cookie. This is a net security improvement as a side effect (creds were previously visible in client JS/devtools) but was done for the session feature, not as an explicit security fix — don't conflate it with "someone asked for a security review."

**Decision**: macOS-style notification behavior — collapsed stack's "×" clears the whole group, expanded individual card's "×" clears just that one. This exact asymmetry was explicitly specified by the user (not my invention) — if it ever seems "wrong" or inconsistent to a future Claude, it's intentional, matching real macOS Notification Center behavior. Don't "fix" it to be symmetric.

`useToast()`'s public signature (`showToast(message, type)`) was deliberately kept unchanged during the stacking rewrite so no existing call site needed touching — if you need a new toast variant/behavior, prefer extending the internal `ToastProvider` logic over changing this signature, to avoid a cascading edit across every page that calls it.

## How I verify UI changes (reusable recipe)

This project has no test suite. Verification has been: `tsc --noEmit` + `eslint` (catches regressions/type errors) plus **actually driving the running app** via Chrome DevTools Protocol, since there's no Puppeteer/Playwright installed (deliberately not added — see "no new dependencies" above) and headless Chrome + the CDP is already possible with zero new installs (`ws` is already a dependency; Chrome itself is present on this machine at `C:\Program Files\Google\Chrome\Application\chrome.exe`).

Recipe used repeatedly this session:
1. Start Chrome headless with remote debugging: `chrome.exe --headless=new --disable-gpu --remote-debugging-port=9222 --user-data-dir=<scratch dir> about:blank`
2. Open a new tab via `PUT http://localhost:9222/json/new?<url>` (must be PUT, not GET, or Chrome returns a text error instead of JSON — bit me once this session).
3. Connect a `ws` client to the tab's `webSocketDebuggerUrl`, send CDP commands (`Page.navigate`, `Runtime.evaluate`, `Page.captureScreenshot`, `Emulation.setDeviceMetricsOverride` for mobile viewport testing) as JSON-RPC, correlate responses by `id`.
4. **Important gotcha**: setting a `<input>`'s `.value` directly via `Runtime.evaluate` does NOT update React's controlled-component state — React won't see the change and native `required` validation blocks submission with "Please fill out this field." You must use the native value setter + dispatch a real `input` event:
   ```js
   const nativeSetter = Object.getOwnPropertyDescriptor(window.HTMLInputElement.prototype, 'value').set;
   nativeSetter.call(el, 'value'); el.dispatchEvent(new Event('input', { bubbles: true }));
   ```
5. Screenshots come back base64 in `Page.captureScreenshot`'s response — write with `Buffer.from(data, 'base64')`. **Use forward slashes in the output path even on Windows** — backslash-escaped Windows paths passed through Bash's quoting to Node got mis-resolved once this session (`C:\Users\...` silently became `C:\tmp\...`), forward slashes avoided the whole problem.
6. Always clean up: kill the headless Chrome process (`taskkill //F //IM chrome.exe`), and clean up any test data written to Postgres during verification (I created and deleted several throwaway restaurants/orders this session — always delete them after, don't leave test rows in the dev DB).

This recipe is worth reusing rather than reaching for a testing library, given the "minimal dependencies, easy to maintain" constraint on this project.

## 6. Concurrency/scaling verification (2026-07)

User asked to confirm Postgres and WebSockets "actually work" and whether multiple simultaneous DB read/writes are possible if scaled. Rather than answer from memory/architecture docs alone, I ran live tests:
- Fired 10 truly concurrent `INSERT`s through the app's `pg.Pool` (default `max: 10`) — all 10 landed successfully in ~40ms, confirming the pool actually parallelizes rather than serializing requests.
- Connected two separate raw `ws` clients to `/ws`, triggered a real order creation through the live API, confirmed **both** clients received the `order_updated` broadcast — confirms the pub/sub fan-out genuinely reaches multiple simultaneous viewers (e.g. a kitchen dashboard + a customer tracker open at once), not just a single hardcoded client.
- Answer given: Postgres concurrency and WS fan-out both already work correctly *within a single running instance* — this required no code changes, just verification. The one real caveat (already documented in SYSTEM_MEMORY.md §8 and above): the WS client registry is in-process memory (`globalThis`), so it would NOT fan out across multiple *separate server instances/processes* (e.g. horizontal scaling behind a load balancer) — that would need Redis or Postgres `LISTEN/NOTIFY` pub/sub, not built and not needed unless the user actually asks to run multiple instances.
- **Lesson for future Claude**: when asked "does X actually work," prefer a live verification (spawn a script, hit the running server, check DB rows) over restating what the architecture docs say should be true. Docs can be stale or aspirational; a live test in this session is ground truth. The `pg`/`ws` packages are already installed, Postgres is already running via Docker — there's essentially no cost to just testing directly.

## 9. Security fix pass, startup/export tooling, WS broadcast scoping (2026-07-05)

### All 11 logged vulnerabilities fixed (F1–F11)
Every finding in `SECURITY_ATTACK_LOG.md` from both attack passes got fixed and live-verified in one session (re-ran each original exploit against the running app, confirmed it no longer works, confirmed the legitimate flow still does). Full details of each fix and its verification are in the log's "Fix pass (2026-07-05)" section — highlights worth knowing without re-reading the whole log:
- **F1/F2 (the big one)**: real random `SESSION_SECRET` generated and set in `.env.local`; `session.ts` now warns loudly on startup if it's ever unset again instead of silently falling back.
- **F5**: added `escapeLikePattern()` (`src/lib/validate.ts`) everywhere user input reaches `ILIKE`; also added an auth guard to `orders/restaurant/[restaurantName]/route.ts`, which previously had none at all.
- **F6/F9**: added `requireString()` (same file) — type + 200-char-length validation — across every free-text input field. Previously a number/object/boolean could silently coerce into `order_number` and get stored, or 500 on an array.
- **F10**: order status changes are now forward-only (`received → preparing → complete`, no skipping/reverting) unless the caller is admin, which can still force any transition — mirrors the existing "God Mode override" precedent elsewhere in the app. Don't revert this to "any transition allowed" without checking with the user first; it was an explicit adversarial finding.
- **F3/F8**: `server.js` now overwrites the client-supplied `X-Forwarded-For` header with the real socket address before Next ever sees it (closes the rate-limiter bypass); `register` route got the same rate limiter the two login routes already had.

### WebSocket broadcast scoping (F7, rebuilt same day after the initial fix)
The first F7 fix (reject `/ws` upgrades with no `Origin` header, cap connections per IP) was flagged by the user as "a speed bump, not full auth" — correctly, since a non-browser client can just set a matching `Origin` itself. Real fix: **the customer tracker is anonymous by design** (no login exists to gate a WS handshake behind — that's the point of "track your order with no account"), so instead of authenticating the socket, each connection now must declare which restaurant it's tracking via `?restaurant=<name>` on the `/ws` URL, and `src/lib/ws-hub.ts`'s `broadcast()` only delivers events to sockets subscribed to the matching restaurant (case-insensitive), instead of blasting every connected socket with every restaurant's order data. `orders/[id]/route.ts`'s PUT/DELETE broadcasts had to be updated to actually include `restaurant_name` in their payload — they didn't before (only the create path did) — since `broadcast()` now drops any event missing that field rather than falling back to "send to everyone."

**If you touch `ws-hub.ts` or `server.js`'s upgrade handler again**: the client registry changed shape from `Set<WebSocket>` to `Set<{ws, restaurantName}>` — don't revert to the flat shape, that's exactly what re-opens the eavesdropping hole. And don't add a new `broadcast()` call site without passing `restaurant_name` in the payload — it'll silently drop the event (fail-closed by design, but worth knowing so it doesn't look like a mystery bug).

**Origin check is still in place too** (kept from the first fix) — belt-and-suspenders, not redundant: it stops a random *website* a victim visits from opening a WS connection at all; the restaurant-scoping stops even a non-browser client that gets past the Origin check from seeing more than the one restaurant it already asked about.

### `startup` and `export` commands added
Two new user-facing entry points, each runnable as `.\startup` / `.\export` from **either** `Restaurant/` or `Restaurant/app/` (thin `.ps1` wrappers in both locations dot-source the real logic in `scripts/`, so there's exactly one copy of each to maintain). Both also have a `.cmd` twin so they work even if a machine's PowerShell execution policy would otherwise block `.ps1` files (`-ExecutionPolicy Bypass` only applies to that one launched process, doesn't touch any system setting).

- **`startup`** (`scripts/startup.ps1`): verbose, step-by-step dependency check — Node, npm, Docker (installed + running), `.env.local` (created + `SESSION_SECRET` generated if missing), then **verifies every declared npm dependency actually resolves** (checks each `node_modules/<pkg>/package.json` exists, not just whether the folder exists or its timestamp looks fresh), reinstalling if anything's missing/broken, then brings up Postgres and starts the dev server. That dependency check isn't hypothetical caution — testing this script on this exact machine found `node_modules` existing but **20 of 20 declared packages** (including `next` itself) missing/broken underneath it; a naive timestamp check would have said "looks fine." If you're tempted to simplify that check back to a timestamp comparison, don't — it was already too weak once.
- **`export`** (`scripts/docker-export.ps1` + `scripts/export-templates/`): builds a Docker image of the app (`app/Dockerfile` — mirrors local dev exactly, `npm run build` then `node server.js`, NOT `next start`, for the same custom-server-needs-raw-WS-upgrade reason as always), bundles it with the Postgres image, a compose file, and one-click `run.cmd`/`run.sh` launchers into `restaurant-app-export.zip` at the repo root. **Both Docker images are saved into the bundle** (not just the app one) specifically so a target machine needs zero internet access to run it — this was a real gap caught during testing (the compose file references `postgres:16` by tag, and Docker would otherwise silently try to pull it from Docker Hub on first run on a machine that's never used Postgres before). See "Docker export self-sufficiency" note below for the full breakdown of what is/isn't needed on the target machine.
- **Real bug found and fixed while testing `export`'s launcher**: the auto-generated `SESSION_SECRET` used `[System.Security.Cryptography.RandomNumberGenerator]::Fill()`, a .NET 6+-only API that fails silently as "method not found" on Windows PowerShell 5.1 (which ships by default on many Windows installs) — the failure didn't stop the script, so it wrote out an **all-zero-byte (entirely predictable) "secret"** as if it were random. Fixed by switching to `RNGCryptoServiceProvider` (compatible with both PS 5.1 and 7+) and adding a hard validation check (empty or all-zero result → fail loudly, don't continue) so this failure mode can never again produce a silently-bad secret. Applied the equivalent guard to the Mac/Linux `run.sh` too (empty/short result from `node`/`openssl` → fail loudly).
- Both scripts were tested against real failure modes, not just the happy path — this entry (not the security findings section) is the authoritative record of what was actually broken and fixed here, since these are bugs in the *tooling*, not security findings against the *app*.

### Docker export self-sufficiency (answering "how self-sufficient is this, really")
On the **target** machine (the one receiving the exported zip), the ONLY prerequisite is Docker Desktop installed and running. Confirmed via direct testing (removed the local image cache, loaded purely from the bundled `.tar` files, ran the actual launcher script, tore it down again):
- No Node.js needed (the Windows launcher's SESSION_SECRET generation uses PowerShell, not Node; the app itself runs from the pre-built image).
- No copy of this source repo needed.
- No Docker Hub / registry account needed.
- **No internet connection needed**, as of the fix described above — both `restaurant-app:latest` and `postgres:16` ship inside the zip as `.tar` files and get `docker load`-ed locally.
- The only thing NOT bundled, by design: existing restaurant/order data. A fresh export always starts the target machine with an empty database — this is a deployment/distribution tool, not a backup/migration tool. If the user ever asks for "also bring my existing data along," that's a different, additive feature (e.g. a `pg_dump` step) — don't assume it's already covered.

### `unpack` command added (same session, after `startup`/`export`)
A third entry point, `.\unpack` / `.\unpack.cmd`, for unpacking `restaurant-app-export.zip` **on the machine that already has this repo** — extracts the zip, runs `docker load` on both bundled images, and generates a fresh `.env`. `-Start` also runs `docker compose up -d` to fully launch it in one command. This exists purely as a same-machine convenience/testing shortcut — a genuinely different machine receiving the zip doesn't have this script and just unzips normally + runs `run.cmd`/`run.sh`, which do the same load+launch steps standalone.

**Real bug found and fixed while testing this**: `docker compose up` failing partway through (e.g. port 3000 already taken by a normal dev server running on the same machine — the exact scenario hit during testing) left orphaned containers/network/volume behind with no cleanup. Fixed: on failure, `unpack` now tears down everything that specific run just created and tells the user the likely cause. A related cosmetic bug was fixed alongside it: the cleanup step's own error output was wrapped in a misleading `NativeCommandError` on PowerShell 5.1 due to redirecting a native command's stderr (`2>&1 | Out-Null`) — looked like the cleanup itself had failed even when it succeeded. Fixed by not redirecting stderr there.

## 13. Cross-platform scripts: Mac/Linux `.sh` equivalents + Windows PS 5.1 hardening (2026-07-05)

User's ask: the `startup`/`export`/`unpack` commands only existed as PowerShell (`.ps1`/`.cmd`), so anyone on Mac or Linux — including a friend cloning the repo per the Option B distribution path documented in USER_HELP.md — had no working entry point at all. Also asked to harden the Windows side and make error messages OS-aware (suggest the right package-manager command instead of a generic list).

**Decision, confirmed with the user via AskUserQuestion first**: write three genuinely independent bash implementations (`scripts/startup.sh`, `scripts/docker-export.sh`, `scripts/docker-unpack.sh`) rather than a single shared Node.js implementation with thin per-platform launchers. The Node-based single-implementation approach was proposed as lower-maintenance, but the user explicitly chose to keep native per-platform scripts and take on the sync burden themselves ("i'll make sure to keep it in check... before ending old convo, i'll have each file updated"). **This means the `.ps1` and `.sh` versions of each command are NOT generated from a shared source and WILL drift if only one is edited** — if you're asked to change behavior in one, check whether the other needs the identical change too, but don't assume the user has already done so or that keeping them in sync is automatically your job across sessions; that's been explicitly delegated to the user.

**What was added**: `scripts/{startup,docker-export,docker-unpack}.sh`, mirroring each `.ps1`'s exact step order and behavior, plus `.sh` wrapper files at repo root and `app/` (matching the existing `.ps1`/`.cmd` wrapper pattern). All made executable (`chmod +x`) since that bit matters for `./script.sh` to work without `bash script.sh` on real Mac/Linux (Windows/Git Bash doesn't enforce it the same way, so this was easy to miss without testing on the actual target).

**Real bugs found and fixed via actual testing in Git Bash** (this machine's closest available bash environment — genuinely exercises real POSIX semantics, unlike a pure Windows shell):
- **POSIX-path-into-`require()` bug (startup.sh)**: the dependency-check step originally built a Node one-liner with `require('$PACKAGE_JSON_PATH')`, where `$PACKAGE_JSON_PATH` was a Git-Bash-style path (`/c/Users/...`). Node's Windows build cannot resolve that path shape via `require()`, even though the shell itself handles it fine — this crashed the whole script every time. Fixed by switching to `fs.readFileSync(process.argv[1], 'utf8')` + `JSON.parse` and passing paths as `process.argv` arguments instead of interpolating them into the JS source string (this also incidentally closes a latent quoting/injection risk from string-interpolating a path into inline JS). **If you add another inline `node -e` call anywhere in these scripts, pass dynamic values via `process.argv`, never interpolate a shell variable directly into the JS source.**
- **Stale RNG bug in `startup.ps1` itself**: the exact `[RandomNumberGenerator]::Fill()` bug already fixed in `run.cmd`/`docker-unpack.ps1` during an earlier session was still present, unfixed, in **`startup.ps1`'s own two `SESSION_SECRET`-generation call sites** — it had been fixed in the exported-bundle launcher but never back-ported to the dev-facing script that has the identical code. Fixed by extracting a shared `New-SessionSecret` function (using `RNGCryptoServiceProvider`, with the same empty/all-zero validation guard) and replacing both inline call sites with it. **Confirmed exploitable and then confirmed fixed by actually running `startup.ps1` under real PowerShell 5.1 on this machine** (not 7+, which wouldn't have shown the bug) — blanked `SESSION_SECRET` in `.env.local`, ran the script, verified a real random value came back, not all-zero. **Lesson**: when the same buggy pattern is fixed in one script, grep the other scripts for the identical pattern before assuming it's fully closed — this bug existed in two places and only one had been patched.
- **Missing `zip`/`unzip` handled gracefully, not just via a crash**: `docker-export.sh` checks for `zip` before attempting to use it (this Git Bash environment doesn't have it installed) and, if missing, still leaves the fully-assembled bundle folder intact and tells the user exactly how to install it or hand off the folder unzipped instead — confirmed by actually running the full export with `zip` absent and verifying both image `.tar`s + all bundle files were present despite the zip step failing. `docker-unpack.sh` has the equivalent check for `unzip`.

**OS/package-manager detection added** (`suggest_install_command()` function, duplicated identically in `docker-export.sh` and `docker-unpack.sh`, plus a lighter inline version in `export-templates/run.sh`): detects `Darwin` (macOS, suggests `brew`) vs `Linux` (checks for `apt`/`dnf`/`yum`/`pacman`/`apk`/`zypper` in that order and gives the one that's actually present) vs `MINGW*`/`MSYS*`/`CYGWIN*` (Git Bash on Windows, suggests MSYS2 `pacman` or Chocolatey) via `uname -s`. Used for the `zip`/`unzip`/`openssl` missing-tool messages so far — if you add a new "please install X" message anywhere in the `.sh` scripts, use this function rather than writing another generic multi-OS instruction list.

**Full end-to-end verification performed** (not just syntax checks): `startup.sh` was run for real against this machine's actual Docker/Node — correctly failed with a clear message when Docker wasn't running, then correctly completed the entire flow (dependency checks → Postgres up healthy → dev server responding 200) once Docker was available. `export.sh` was run for real, producing genuine `restaurant-app-image.tar` (252MB) and `postgres-image.tar` (159MB) inside the bundle folder. `unpack.sh` (both with and without `--start`) was run for real, including removing the locally-cached `restaurant-app:latest` image first to simulate a clean machine, confirming `docker load` + `docker compose up` brought up a working stack reachable on a non-default port. All test containers/networks/volumes were torn down afterward; the pre-existing dev environment was left in its prior state.

The user separately opened Docker Desktop themselves around the same time and, when told about the auto-launch, said explicitly: "although i don't mind you starting it up on your own either." **This supersedes the older blanket rule below** — auto-starting Docker Desktop programmatically is now fine, not just tolerated. ## 14. Error handling/resilience audit, performance pass, status-text bugfix, health pin (2026-07-05)

User's ask: audit the whole app for error handling (correctly guessed there wasn't much — a dropped DB connection or flaky request had no retry/fallback anywhere), add graceful degradation/retry/fallback behavior everywhere it made sense, then do a low-effort performance pass, explicitly asking me not to burn tokens on it. Full technical detail is in `SYSTEM_MEMORY.md` §14 — this entry is the judgment-call/narrative record.

**Scope discipline**: I read every relevant file (routes, db layer, ws-hub, server.js, every client page that fetches) before writing anything, then fixed real, concrete gaps rather than adding defensive code everywhere reflexively — e.g. I did NOT add retry to order creation or restaurant registration (both use `{ retries: 0 }` in the new `fetchJson`/`fetchWithRetry` helper) because those aren't idempotent from the client's perspective; blindly retrying a "timed out but maybe actually landed" create/register request risks a confusing 409 rather than a silent duplicate, so those get a clean single-attempt error message instead of a transparent retry. This distinction (idempotent-safe-to-retry vs. not) is the operating principle for every retry decision made this session — if you add a new retry call site, ask that question first rather than defaulting either way.

**Real, previously-invisible crash risk found and fixed**: the `pg.Pool` had no `'error'` listener. An idle pooled client emitting a background error (DB restart, network blip, no request in flight to catch it) is an *unhandled* Node event, which is fatal — it would have silently killed the entire server process, not just that one connection. This had never been hit in practice (hence never noticed), but it's exactly the kind of gap that turns "Postgres blipped for 2 seconds" into "the whole app is down and needs a manual restart." Fixed with a listener that logs and lets the pool self-heal, plus `process.on('uncaughtException'/'unhandledRejection', ...)` in `server.js` as a second layer of the same protection.

**Performance win, not just error handling**: `initDb()` (6 round-trip `CREATE ... IF NOT EXISTS` queries) was being awaited on literally every mutating request — order create/update/delete, restaurant register/delete/password-reset. Since nothing in this codebase ever drops a table (purge is `DELETE FROM`, not `DROP TABLE`), it's a pure no-op after the first successful run per process. Memoized it — same behavior, 6 fewer round-trips per request after the first. Also found and fixed a real wasted-poll bug in God Mode (`admin/page.tsx`): it polled the *entire* restaurants+orders dump every 5s regardless of which sim view was active or whether the tab was even visible — gated it on `simMode === "ADMIN"` and `document.visibilityState === "visible"`, a genuine load reduction with zero visible behavior change (nothing was ever shown from the wasted polls anyway). **Deliberately did NOT apply the same visibility-gating to the Kitchen Dashboard's own 5s poll** — that's the actual working screen, and skipping polls while backgrounded risks visibly stale orders the moment kitchen staff tab back in; not the same trade as God Mode's simulation view.

**Bug the user remembered mid-session and asked me to fix before touching the docs**: kitchen sets an order to "Preparing," customer tracker shows "Ready for Pickup!" — color/icon were correct, only the text was wrong. Root cause: `customer/page.tsx`'s title/description logic did raw `===` string comparison against the *customer* vocabulary (`Received/Making/Finished`), but `order.status` from the API is always the *Kitchen* vocabulary (`Received/Preparing/Complete` — this is the long-documented status-vocab-inconsistency quirk in `SYSTEM_MEMORY.md` §2, still real). `"Preparing"` matched none of the three raw string branches and fell through to the final `else`. The color/icon *were* already right because those specifically route through `getStatusVisual()`→`normalizeStatus()`, which already unifies both vocabularies — the title/description logic just never got the same treatment when that unification layer was built. Fixed by keying both off `normalizeStatus(order.status)` instead of literal strings, and found + fixed the identical bug class in the same file's WS-reconnect-effect exit condition (`order.status === "Finished"` never matched a `"Complete"`-vocabulary order, so the socket would never self-close for a genuinely finished order). **If `customer/page.tsx` is touched again: never compare `order.status` via raw `===` against a literal status string anywhere in that file — always go through `normalizeStatus`/`getStatusVisual`, or this exact bug silently reappears.** Verified live: created a real order via the API, moved it to "Preparing" via `PUT /api/orders/[id]`, drove the actual rendered customer page via headless Chrome (same CDP recipe as prior sessions), confirmed it now reads "In the Kitchen" / the correct description instead of "Ready for Pickup!".

**Feature added same session, user's idea mid-task**: a small health-status pin for God Mode (`/admin`), top-right, healthy/ok/bad/terrible tiers. Backed by a new admin-gated `GET /api/health` that runs a live `SELECT 1` for real DB latency, reads the `pg.Pool`'s own `totalCount`/`idleCount`/`waitingCount`, and reads the WS hub's live client count straight off the shared `globalThis` registry — no new state to maintain, all signals were already sitting there unexposed. Tier thresholds: `terrible` = the DB query itself failed (not just slow), `bad` = pool has callers waiting for a connection or latency > 300ms, `ok` = latency > 50ms, `healthy` = otherwise. Polls every 10s, paused when the admin tab isn't visible (same reasoning as the God Mode data-poll gating above). Verified live: logged into the actual admin UI via headless Chrome, confirmed it renders "Healthy" with the real Postgres container reporting ~14ms latency, and confirmed `/api/health` correctly 401s when unauthenticated.
